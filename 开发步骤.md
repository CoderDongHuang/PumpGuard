## Phase 1：仿真数据引擎

> **一致性检查**：与 [项目架构说明书](./项目架构说明书.md) 第六章完全对齐。

### 1.1 水泵物理模型

**输入参数**：
- 泵型号（离心泵/轴流泵/混流泵）
- Q-H-η 特性曲线参数（额定流量、额定扬程、额定效率、BEP 点位置）
- 工况参数（转速、负载率、入口压力）

**输出**：
- 正常工况下的理论振动 RMS、理论效率、理论温升
- 退化趋势：振动缓升（指数型）、效率缓降（线性型）、温度缓升（指数型）

**关键文件**：
```
simulation-engine/
├── models/
│   ├── pump_physics.py      # Q-H-η 曲线建模
│   ├── degradation.py       # 退化动力学模型
│   └── fault_injection.py   # 故障注入引擎
├── sensors/
│   ├── vibration.py         # 振动频谱仿真（基频+谐波+噪声）
│   ├── current.py           # 电流波形仿真（含故障特征频率）
│   └── thermal.py           # 温度仿真
└── engine.py                # 仿真引擎主入口
```

### 1.2 正常退化建模

1. 基于 Q-H-η 曲线计算各工况点的理论参数
2. 加入退化漂移（振动每年 +5-15%、效率每年 -1-3%、温度每年 +2-5°C）
3. 加入传感器噪声（高斯噪声 SNR 30dB）
4. 加入工况波动（负载率 ±10% 随机变化）

### 1.3 故障注入

| 故障类型 | 注入方式 | 仿真信号特征 |
|---------|---------|-------------|
| 轴承内圈磨损 | 振动频谱增加 BPFI 频率分量 | BPFI = 0.6 × N × RPM |
| 轴承外圈磨损 | 振动频谱增加 BPFO 频率分量 | BPFO = 0.4 × N × RPM |
| 叶轮气蚀 | 振动频谱 5-20 kHz 频段能量增加 | 宽频噪声 + 冲击脉冲 |
| 不对中 | 振动频谱 1×/2× 基频分量显著增加 | 1× + 2× RPM 频率 |
| 密封泄漏 | 出口压力持续下降 + 流量下降 | 无频谱特征，趋势型 |
| 电机电气故障 | 电流频谱增加槽通过频率 + 谐波畸变 | 电流 THD 上升 |

### 1.4 输出格式

每次仿真输出 JSON 行（JSONL）格式，字段与架构说明书 3.1.4 统一数据模型一致。

### 1.5 参考数据集集成

- **NASA IMS Bearing**：下载 → 解析 → 格式化为统一数据模型 → 作为真实退化参考
- **Kaggle Digital Twin**：下载 → 对齐字段 → 与自身仿真引擎输出对比验证

## Phase 2：数据接入层

> **一致性检查**：与 [项目架构说明书](./项目架构说明书.md) 3.1 节完全对齐。

### 2.1 MQTT Broker 搭建

1. 部署 EMQX（开源 MQTT Broker）
2. 配置 Topic 命名规范：`pump/{device_id}/{data_type}`
3. 仿真引擎 → MQTT（模拟新泵直采数据流）

### 2.2 边缘网关（Demo 简化版）

1. MQTT Subscriber 接收原始数据
2. 数据清洗：去噪（中值滤波）、插值（线性插值）、异常值检测（3σ 规则）
3. 特征提取：RMS 值、FFT 主频分量、电流谐波畸变率
4. 清洗后数据 → 发布至 MQTT `pump/{device_id}/processed`

### 2.3 数据入库

1. Java 服务订阅 MQTT 处理后的 Topic
2. 写入 InfluxDB（时序数据）+ PostgreSQL（设备元数据同步）
3. 实现 MCSA 模拟数据生成（基于仿真电流波形做 FFT 分析）

### 2.4 边缘网关实际部署考虑

> **⚠️ 与文档的潜在误差**：架构说明书 3.1.3 描述了边缘网关"可运行于工控机或 ARM 边缘设备"。实际 Demo 阶段边缘网关与仿真引擎合并运行在同一机器，不做真正的边缘分离部署。这不影响演示效果，但需在产品化时补充真正的边缘部署方案。

### 3.2 HI 计算引擎实现

1. **物理基线构建**：
   - 读取设备台账中的 Q-H-η 曲线参数
   - 根据当前工况（转速、流量）查表获取理论振动/效率/温度
   - 计算各维度的偏离度

2. **HI 融合公式**（初始版本采用加权平均，后续可替换为学习型融合）：

   ```
   HI = 100 - (W_v × ΔV_score + W_η × Δη_score + W_T × ΔT_score + W_t × t_score)
   
   其中各 W 初始值：W_v=40, W_η=30, W_T=20, W_t=10（总和 = 100）
   各 Δ_score 归一化到 0-100
   ```

3. **输出格式**：HI 值（0-100）+ 各子维度得分 + 退化速率（HI 变化率/天）

### 3.3 故障预测实现

1. **机理规则层**（优先实现）：
   - 振动 RMS 超过 ISO 10816 标准 → 不对中/不平衡
   - 轴承特征频率分量超过基线 6dB → 轴承故障
   - 叶轮通过频率分量异常 → 叶轮故障
   - 电流 THD 突增 → 电机电气故障

2. **LSTM 模型**（后续实现）：
   - 滑动窗口：前 30 天数据 → 后 7 天故障概率
   - 训练数据：仿真引擎生成 150 台泵 × 多种故障 × 多工况

3. **联合判定**：
   - 机理规则输出：`{fault_type, probability_rule}`
   - LSTM 输出：`{fault_type, probability_dl}`
   - 融合：`probability = max(probability_rule, probability_dl)`，若两者分歧则降置信度

### 3.4 RUL 预测实现

1. **趋势外推法**（优先实现）：
   - 取最近 30 天 HI 序列
   - 指数拟合 / 线性拟合外推至失效阈值（HI < 30）
   - 输出：预估剩余天数 + 80% 置信区间

2. **相似性匹配**（后续实现）：
   - 将当前泵的 HI 退化曲线与同型号历史退化曲线做 DTW 匹配
   - 取 Top-3 最相似曲线，加权平均 RUL

### 3.5 根因分析实现

1. 建立故障特征库（JSON 配置）：
   ```json
   {
     "bearing_inner_race_wear": {
       "vibration_features": ["BPFI分量 > 基线6dB", "1×RPM谐波增加"],
       "current_features": ["无显著特征"],
       "temperature_features": ["轴承温升 > 基线10°C"],
       "confidence_weight": 0.9
     }
   }
   ```

2. 输入当前特征 → 遍历特征库 → 计算匹配度 → 排序输出

### 3.6 模型训练与评估

1. 仿真引擎生成训练集/验证集/测试集（70/15/15 划分）
2. 训练 LSTM 故障预测模型
3. 记录 MLflow 实验
4. 评估指标：准确率、精确率、召回率、F1、ROC-AUC

### 4.2 工单自动生成与路由

1. 监听算法层告警事件（Kafka Topic: `alert.triggered`）
2. 根据 HI 值 + 故障类型 → 确定紧急程度（P0/P1/P2/P3）
3. 生成工单 → 写入 PostgreSQL
4. 匹配工程师（模拟实现：预设 3-5 个工程师的 skill set + location 数据）
5. 工单创建成功 → Kafka Topic: `workorder.created`

### 4.3 备件库存优化

1. 监听 RUL 预测结果（Kafka Topic: `rul.updated`）
2. RUL < 30 天 → 查询备件库存
3. 库存不足 → 生成备件采购建议
4. 模拟实现：预设备件库（轴承/密封/叶轮 + 库存量）

### 4.4 全球服务网络调度

1. Demo 阶段：预设 5 个服务区域 × 3 个工程师
2. 就近匹配算法：Haversine 距离计算
3. 路径优化：多站点巡检 TSP 简化版（最近邻即可）

## 附录：技术难点与建议

| 难点 | 难度 | 建议 |
|------|------|------|
| LSTM 故障预测模型训练 | 中 | 使用仿真引擎生成充足训练数据；先用 XGBoost 做 baseline |
| HI 融合权重确定 | 中 | 初期使用经验权重（W_v=40, W_η=30, W_T=20, W_t=10），后续可通过回归学习优化 |
| 振动频谱仿真真实性 | 中 | 参考 NASA IMS 数据集频谱特征，确保仿真频谱包含正确的故障特征频率 |
| 飞书 Bot 卡片交互 | 低 | 飞书卡片消息构建器（Card Builder）可视化搭建 |
| Docker Compose 多服务编排 | 低 | 从单个 docker-compose.yml 起步，不做 K8s |
| gRPC proto 编译 | 低 | 两侧（Java/Python）各用官方 protoc 插件 |
