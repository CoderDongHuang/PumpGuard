# 利欧泵业 AI 智慧预测与健康管理（PHM）系统 — 方案规划

## 一、项目本质

构建一套**物理机理与数据双驱动**的水泵智慧预测与健康管理系统，让全球百万台泵从"哑设备"变成"会说话"的智能体——主动报告健康状态、提前预警故障、自动触发运维决策，并在维修反馈中持续自进化。

这不是一道纯粹的算法题，而是一道**工业系统设计题**，核心考验三点：
- 数据怎么采得到（老旧设备兼容）
- 模型怎么信得过（小样本/零故障也可靠）
- 闭环怎么转起来（从预警到维修到模型进化）

---

## 二、完整架构（四层一闭环）

### 1. 数据接入层：有感 + 无感分层策略

| 设备类型 | 采集方式 | 数据内容 | 上云路径 |
|---------|---------|---------|---------|
| 新泵/智能泵 | 原生传感器直采 | 高频振动、温度、电流波形、压力、流量 | MQTT → 物联网平台 |
| 老旧泵（有变频器） | 接入变频器已有数据 | 电流、频率、运行状态 | 边缘网关 → MQTT |
| 老旧泵（无变频器） | 外挂振动温度复合传感器 + 钳形电流表 | 振动、温度、电流特征（MCSA） | 边缘网关 → MQTT |

边缘网关层做协议转换、数据清洗和初步特征提取，上层统一数据模型做归一化映射。**不管新旧，都能"说话"。**

---

### 2. 算法层：机理 + 数据融合驱动（核心竞争力）

这是整个系统的大脑，也是物联网专业背景相对于纯软件选手的绝对优势：

- **健康指数（HI）**：以水泵物理特性（Q-H-η 曲线）为基线，融合实际偏离程度：
  ```
  HI = f(振动偏离度, 效率衰减度, 温升趋势, 运行时长)
  ```
  物理模型给出"理论上这个工况下应该是什么样"，数据模型学习"实际偏差意味着什么"。不纯靠数据驱动，零故障样本也能有基线判断力，不会乱报警。

- **故障预测**：时序模型（LSTM/Transformer）+ 机理规则联合判定，小样本下也能给出可解释的预警。

- **剩余寿命预测（RUL）**：基于 HI 退化趋势外推，结合同型号设备历史退化曲线做相似性匹配。

- **根因分析**：故障特征库 + 因果推理，输出"可能原因 + 置信度 + 建议处置方案"。

**核心优势：物理先验注入模型 = 不乱报警 + 可解释 + 小样本可用。**

---

### 3. 决策层：智能运维调度

- **工单自动生成**：检测到异常 → 自动评估紧急程度 → 生成工单并路由到最合适的工程师（距离、技能、负载）。
- **备件优化**：基于 RUL 预测，提前计算备件需求量和库存分布，减少紧急采购。
- **全球调度**：150+ 国家的服务网络，就近派单 + 技能匹配。

---

### 4. 反馈闭环：系统自进化

```
维修工程师现场确认 → 标记 AI 诊断是否正确 → 反馈回流算法层 → 模型持续更新
```

每修一次泵，模型就聪明一分。解决工业 AI 最致命的痛点——标注数据永远不够。

---

### 5. 展示层

- **数字孪生看板**：全球泵群健康状态一张图，支持从洲际下钻到单台泵。单台泵数字孪生体关联实时传感器数据、历史维修记录、同型号退化基线，鼠标悬停即可查看该泵全生命周期健康档案。
- **飞书 Bot 告警**：异常实时推送，结合飞书 AI 工具做智能问答和运维报告生成。
- **移动端**：巡检助手 + 维修反馈录入。

## 三、技术架构

```
┌─────────────────────────────────────────────────┐
│           展示与协同层（Web看板 + 飞书Bot + 多维表格 + 审批流）              │
├─────────────────────────────────────────────────┤
│      业务主脑：Java Spring Boot                    │
│  设备管理 │ 数据接入 │ 工单流转 │ 权限 │ 告警路由       │
├─────────────────────────────────────────────────┤
│      AI 引擎：Python FastAPI（微服务）              │
│  异常检测 │ RUL 预测 │ 根因分析 │ HI 计算            │
├─────────────────────────────────────────────────┤
│     数据层：时序库（InfluxDB）+ PostgreSQL + 消息队列  │
├─────────────────────────────────────────────────┤
│     边缘层：网关采集 │ 协议转换 │ 特征提取              │
└─────────────────────────────────────────────────┘
```

- **Java Spring Boot**：业务主脑，负责设备管理、数据接入、工单流转、权限管理、告警路由。
- **Python FastAPI**：AI 引擎，封装异常检测、RUL 预测、根因分析等推理接口，作为微服务被 Java 调用。
- 两者通过 REST/gRPC 通信，架构清晰，是大厂级的微服务范式。

## 四、演示方案

用 Python 构建泵群仿真数据引擎，模拟正常退化并注入典型故障，驱动从异常感知、健康评分到工单自动触发的全自动闭环演示。

时间充足的情况下，深度打磨三个亮点：
1. **仿真数据引擎可配置**：支持不同型号泵、不同工况、不同退化模式、不同故障注入，本身就是一个值得展示的子系统。
2. **HI 模型跑出真实退化曲线**：用真实水泵 Q-H-η 曲线数据构造有理有据的仿真退化过程，可视化比任何 PPT 都有说服力。
3. **飞书集成做深**：飞书多维表格做运维台账、飞书审批流做工单闭环、飞书 AI 做自然语言查询（例如"3号线2号泵现在健康度多少？"）。

---

## 五、一句话总结

这是一个**物理机理 + 数据驱动融合**的泵群 PHM 系统，**兼顾新老设备分层接入**，以 Java 为业务骨架、Python 为 AI 引擎，通过**维修反馈闭环实现模型自进化**，最终以仿真数据引擎驱动全自动闭环演示。
